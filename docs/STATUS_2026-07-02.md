#### P1 — Drag&drop NIE DZIAŁA (3 iteracje, nadal zepsute)
- **Symptom:** drop akceptowany (highlight się pokazuje), plik "czyta i znika" —
  nie ląduje na liście tracków / kolejce playera / data disc.
- **Co próbowane:** `.onDrop` na TrackListView, FileDropExtractor z testami,
  usunięcie fileExists filter, `startAccessingSecurityScopedResource()`.
- **Hipoteza残余:** problem może być w (a) samym `.onDrop` hit-testingu w zagnieżdżonym
  NavigationSplitView, (b) loadItem async który nie woła completion handler dla
  Finder drops, (c) `session.addFiles` cicho filtruje po extension, (d) URL-e z Findera
  mają inny format niż testujemy.
- **Działanie jutro:** ręczny debug z print logami w handleDrop + extractURLs, sprawdzenie
  czy `box.urls` jest non-empty po dropie, czy `session.addFiles` realnie dodaje.
  Rozważyć XCUITest drag-drop UI test. MOŻLIWE że `.onDrop` w ogóle nie działa w tym
  view hierarchy i trzeba innej strategii (NSItemProvider trực trên okno przez NSView representable).

#### P2 — Player + Burner concurrency — brak UX info
- **User:** "jak mam utwór w playerze który gra to nie mogę już dodać innego przez
  dodaj plik ani dragndrop" + "czy można słuchać playera i jednocześnie nagrywać
  inną płytę? powinno to być widoczne w menu albo popup"
- **Diagnoza:** brak info w UI że player działa w tle podczas burn, brak separation
  stanu między Player a Burner (współdzielą BurnSession? — sprawdzić).
- **Działanie jutro:** dodać status indicator "▶ Player aktywny" w burnerze / "🔥 Burn
  w toku" w playerze; popup info gdy user próbuje akcji która koliduje; lub thread-safe
  współdzielenie stanu żeby oba mogły działać równolegle.

#### P3 — "Add file" po odtworzeniu pierwszego tracku NIE DZIAŁA w playerze
- **User:** raz odtworzony track → "dodaj plik" / drag&drop nie dodaje.
- **Diagnoza:** `player.queueCount > 0` → gałąź else → `player.queue(url:)`.
  Może `queue(url:)` rzuca silently albo `@Published queueCount` nie triggeruje UI.
  Albo security-scope nie startuje dla drugiego pliku.
- **Działanie jutro:** print logi w `player.queue(url:)` i `handleFileImport` drugim
  przebiegu; sprawdzić czy `queueCount` się aktualizuje.

## 3. Setup orkiestracji — DUAL-TERMINAL (zapisane na przyszłość)

**Zasada:** CEO (glm-5.2, darmowy) orkiestruje w terminalu 1; worker (deepseek-v4-flash,
tani) koduje w terminalu 2. CEO steruje przez `intercom`. Oszczędza tokeny.

### Krok po kroku (uruchamiać na początku każdej sesji roboczej):
1. User otwiera 2. terminal: `cd ~/Desktop/Spalam\ Sie && pi --provider opencode-go --model deepseek-v4-flash`
2. User mówi tam: `Czekaj na zadania od CEO przez intercom. Odpowiadaj na ask.`
3. User wraca do terminala CEO, mówi "gotowe"
4. CEO: `intercom list` (widzi 2 sesje), `intercom ask to:<worker> message:<spec>`
5. Worker koduje TDD RED→GREEN, zwraca wynik
6. CEO: review diff + `swift test` + `bash build-app.sh` → commit
7. Iteracja: max 3x, potem CEO stop+myśli (zmiana testu/podejścia)

### Weryfikacja kosztów
- Gateway opencode-go jest opaque — modelId może być tylko etykieta.
- Po 1-2 taskach user zagląda w panel rozliczeń opencode-go: czy sesja deepseek
  rozliczona tanio. Jeśli drogo → skonfigurować realny provider DeepSeek z kluczem
  z platform.deepseek.com w `~/.pi/agent/models.json`.

### Protokół TDD workera
- RED: failing test + wklej output
- GREEN: implementacja + test pass
- REFACTOR: jeśli trzeba
- FULL suite + build + git diff --stat
- Worker NIE commituje — CEO review + commit

Pełny setup zapisany w context-mode KB (source: `ceo-dual-terminal-setup`).
